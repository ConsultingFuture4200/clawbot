Build the Telegram digest formatting and delivery layer: message formatting with HTML parse mode, batch buffer for smart batching, urgent standalone notifications, and numbered reply handling.

Purpose: Without delivery, classification results exist in memory but never reach the user. This plan closes the feedback loop by surfacing every classified email in Telegram with the correct format, priority routing, and interactive reply handling.

Output: digest-formatter.js (message formatting), delivery.js (buffer management + send orchestration + reply handler), initialized state files.
</objective>

    Create `sandbox/skills/classify-email/digest-formatter.js` (CommonJS module).

    **Function: `escapeHtml(text)`**
    - Replace `&` with `&amp;`, `<` with `&lt;`, `>` with `&gt;`
    - All user-sourced strings (sender, subject, reasoning, recommended_action) MUST be escaped before insertion into HTML templates

    **Function: `formatDigest(classifiedEmails, account)`**
    - Input: array of classified email objects (from Plan 02 classifyPipeline output), account string
    - Returns: array of strings (each <= 4000 chars), formatted as HTML for Telegram `parse_mode: "HTML"`
    - Implementation per 02-UI-SPEC.md copywriting contract:

    Account header: `<b>${account.toUpperCase()}</b> (${count} new)\n\n`
    Continuation header: `<b>${account.toUpperCase()}</b> (continued)\n\n`

    Per-email entry format (construct for each email):
    ```
    {globalNumber}. [{CATEGORY}] {escapedSender}
       "{escapedSubject}"
       -> {escapedRecommendedAction}
    ```

    Modifiers applied per UI-SPEC rules:
    - If `is_urgent === true`: append ` URGENT` to category tag, e.g., `[CODE URGENT]`
    - If `is_unknown_sender === true`: append `\n   NEW SENDER` after action line (D-09)
    - If top category confidence < 0.70 (ask_user tier): prefix action with `? ` and append reasoning in italic:
      ```
      {number}. [{CATEGORY}] {sender}
         "{subject}"
         ? -> {action}
         <i>Reasoning: {escapedReasoning}</i>
      ```
    - Multi-label: show primary category (highest confidence) in tag; if secondary categories exist with confidence > 0.50, mention in action text

    Footer on last message chunk: `\n<i>Reply with a number to act on that email.</i>`

    **Message splitting** (TGRAM-06):
    - MAX_MSG_LENGTH = 4000
    - Before appending each email entry, check if `(current + entry).length > MAX_MSG_LENGTH`
    - If so, append `<i>continued...</i>` to current message, push to messages array, start new message with continuation header
    - Numbering is continuous across splits (globalNumber never resets)
    - Never split mid-entry

    Sort order within each account group: urgent first, then by primary category confidence descending

    **Function: `formatUrgentNotification(classifiedEmail)`**
    - Input: single classified email object with `is_urgent === true`
    - Returns: single string formatted per UI-SPEC urgent standalone template:
    ```
    URGENT EMAIL

    From: {escapedSender}
    Subject: "{escapedSubject}"
    Account: {account}
    Category: {categoryTag}

    -> {escapedRecommendedAction}

    <i>Reply to act on this email.</i>
    ```

    **Function: `formatEmptyState(spamCount)`**
    - If spamCount === 0: return `No new emails since last digest.`
    - If spamCount > 0: return `${spamCount} emails received, all filtered as spam/noise. No action needed.`

    **Function: `formatErrorState(errorType, details)`**
    - errorType: 'gemini_api' | 'ollama_offline' | 'gmail_api' | 'rate_limit'
    - Returns the corresponding error message string per UI-SPEC error states table:
      - gemini_api: `Classification unavailable -- Gemini API error. ${details.count} emails queued for next cycle. Check API key and rate limits.`
      - ollama_offline: `Local spam filter offline -- Ollama not responding. Sending all ${details.count} emails to Gemini for classification.`
      - gmail_api: `Gmail sync failed for ${details.account}. Error: ${details.error_code}. Will retry next heartbeat cycle.`
      - rate_limit: `Gemini rate limit reached. ${details.count} emails queued for classification in next cycle (${details.minutes} min).`

    Export: `module.exports = { formatDigest, formatUrgentNotification, formatEmptyState, formatErrorState, escapeHtml };`

    Constants:

    - `const BATCH_INTERVAL_HOURS = 3;` (TGRAM-03: 3-hour batch cycle)
    - `const DIGEST_EXPIRY_HOURS = 24;` (UI-SPEC: replies to digests >24h old are expired)

    **Function: `addToBatchBuffer(classifiedEmail)`**
    - Loads batch-buffer.json
    - Appends email to `buffer[classifiedEmail.account].emails` array
    - Writes updated buffer back to disk
    - Email object stored in buffer includes: threadId, messageId, sender, subject, snippet, classification (full object), is_unknown_sender, account, received_at (current timestamp)

    **Function: `shouldSendDigest(account)`**
    - Loads batch-buffer.json
    - Returns `true` if: `buffer[account].emails.length > 0 AND (last_digest_sent is null OR hours since last_digest_sent >= BATCH_INTERVAL_HOURS)`

    **Function: `sendDigest(account, telegramSendFn)`**
    - Input: account string, telegramSendFn = `async (text, parseMode) => telegramMessageId` (OpenClaw Telegram channel callback)
    - Loads batch-buffer.json
    - Gets emails for account, sorts: urgent first, then by primary confidence descending
    - Calls `formatDigest(emails, account)` to get array of message strings
    - For each message string, calls `telegramSendFn(message, 'HTML')` and collects returned Telegram message IDs
    - Builds digest map entry: for each email, maps its `globalNumber` to `{ threadId, messageId, account }`
    - Stores mapping in digest-map.json keyed by first Telegram message ID:
      ```json
      { "telegram_msg_ID": { "sent_at": "ISO timestamp", "mapping": { "1": {...}, "2": {...} } } }
      ```
    - Clears buffer for account: set `emails: []` and `last_digest_sent: new Date().toISOString()`
    - Writes updated batch-buffer.json
    - Returns: number of emails delivered

    **Function: `sendUrgentNotification(classifiedEmail, telegramSendFn)`**
    - Input: single classified email with delivery='immediate', telegramSendFn callback
    - Calls `formatUrgentNotification(classifiedEmail)` to format message
    - Calls `telegramSendFn(message, 'HTML')` and gets Telegram message ID
    - Stores single-entry mapping in digest-map.json for reply handling (entry key = telegram msg ID, mapping = `{ "1": { threadId, messageId, account } }`)
    - Returns: Telegram message ID

    **Function: `handleDigestReply(telegramMsgId, replyText)`**
    - Input: Telegram message ID being replied to, the reply text
    - Loads digest-map.json
    - Finds the digest entry for telegramMsgId
    - If not found: return `{ error: 'no_mapping', message: 'Could not find the referenced digest.' }`
    - If found but `sent_at` is > DIGEST_EXPIRY_HOURS ago: return `{ error: 'expired', message: 'This digest has expired. Check the latest digest for current emails.' }`
    - Parses replyText: extract number (first integer in string)
    - If not a valid number: return `{ error: 'non_numeric', message: null }` (silently ignored per UI-SPEC)
    - If number not in mapping range: return `{ error: 'out_of_range', message: 'No email #${number} in the last digest. Valid range: 1-${maxNumber}.' }`
    - If valid: return `{ success: true, email: mapping[number] }` with the threadId, messageId, account for the selected email

    **Function: `processClassifiedEmails(classifiedResults, telegramSendFn)`**
    - Input: output from classifyPipeline `{ classified: [...], failed: [...] }`, telegramSendFn
    - For each classified email:
      - If `delivery === 'immediate'`: call `sendUrgentNotification(email, telegramSendFn)`
      - If `delivery === 'batch'`: call `addToBatchBuffer(email)`
    - After processing all emails, check both accounts: if `shouldSendDigest('work')`, call `sendDigest('work', telegramSendFn)`. Same for 'personal'. (Work first per Phase 1 D-12 / UI-SPEC account order)
    - For failed emails: log warning, they will be retried next heartbeat cycle
    - Return: `{ urgent_sent: N, buffered: N, failed: N, digests_sent: ['work'|'personal'] }`

    **Digest map cleanup:** In `handleDigestReply` or at digest send time, prune digest-map.json entries older than 48 hours to prevent unbounded growth.

    Export: `module.exports = { addToBatchBuffer, shouldSendDigest, sendDigest, sendUrgentNotification, handleDigestReply, processClassifiedEmails };`
