# Phase 2: Classification & Delivery - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Every incoming email is classified into 7 categories (code, calendar, research, home, urgent, routine, spam/noise) with confidence scores using Gemini structured JSON output, and surfaced to the user via structured Telegram digests with smart batching (urgent = immediate, low-priority = every 3 hours). This phase delivers classification + delivery as one vertical slice — classification without delivery is untestable.

</domain>

<decisions>
## Implementation Decisions

### Confidence Thresholds
- **D-01:** Three tiers: >0.85 auto-label (shown in digest, no prompt), 0.70-0.84 act-and-confirm (classification shown with 'correct?' flag, silent confirm if user ignores), <0.70 ask user (inline in digest with '?' marker and agent's best guess, user replies with correct category)
- **D-02:** Urgent emails ALWAYS trigger an immediate standalone Telegram notification regardless of confidence score — better to over-alert than miss
- **D-03:** Mid-confidence (0.70-0.84) items that are ignored by user are treated as implicitly confirmed — no blocking

### Classification Pipeline
- **D-04:** Two-stage pipeline: local ollama/qwen2.5:7b runs binary spam gate first ("is this obviously spam/noise?"), only non-spam emails proceed to Gemini for full 7-category classification
- **D-05:** Gemini classifies in batches of 5 emails per prompt call (metadata only: sender, subject, snippet), using structured JSON output with response schema
- **D-06:** Few-shot examples maintained as a static, manually curated JSON file (classification-examples.json) with 2-3 examples per category (14-21 total). Updated occasionally by user, not auto-growing.
- **D-07:** Chain-of-thought reasoning generated by Gemini during classification. Internal/logged for debugging by default, but reasoning summary shown in Telegram digest for low-confidence (<0.70) items only

### Unknown Sender Handling
- **D-08:** "Unknown" = first-time sender, never seen before in that account's email history. Determined via a sender cache built from history.list data over time.
- **D-09:** Unknown senders flagged in digest with a visual marker (e.g., "⚠️ New sender") but otherwise appear normally with their classification and priority
- **D-10:** Sender caches are separate per account (personal and work) — respects account separation established in Phase 1 (D-11, D-14). A sender known to work but not personal still shows as 'New sender' in personal digests.

### Telegram Digest Format
- **D-11:** Claude's discretion on digest layout, grouping, numbered selection UX, and message splitting for long batches. Must satisfy TGRAM-01 through TGRAM-06 requirements.

</decisions>

<specifics>
## Specific Ideas

- Chain-of-thought reasoning shown to user only when confidence is low (<0.70) — keeps digests clean while giving transparency where it matters
- Ollama spam gate is intentionally lightweight ("is this obviously spam?") — not a full classifier, just a binary filter to save Gemini quota
- Sender cache builds organically from history.list data — no manual seeding or Google Contacts dependency needed
- "Ignored = confirmed" for mid-confidence items — reduces friction while still giving user the chance to correct

</specifics>
