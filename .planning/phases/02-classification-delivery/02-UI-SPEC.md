# Phase 2 — UI Design Contract

> Visual and interaction contract for Telegram-based email triage digest delivery. Generated by gsd-ui-researcher.

**Important context:** This project has no web frontend. The sole user interface is Telegram Bot API messages. All visual styling is constrained by Telegram's rendering engine -- no custom fonts, colors, spacing, or CSS. This contract defines the **message format, interaction patterns, and copywriting** that constitute the "UI" for Phase 2.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none (Telegram Bot API) |
| Icon library | Unicode emoji (Telegram-native) |
| Font | Telegram default (platform-controlled, not customizable) |

**Rendering engine:** Telegram Bot API `sendMessage` with `parse_mode: "HTML"`
**Rationale:** HTML parse mode chosen over MarkdownV2 per RESEARCH.md Pitfall 7 -- HTML requires escaping only `&`, `<`, `>` vs MarkdownV2's 19+ special characters. Source: 02-RESEARCH.md Pattern 4.

---

## Spacing Scale

Telegram controls all spacing. The following logical spacing is achieved through newlines and whitespace characters:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 0 lines | Inline elements within a single digest entry |
| sm | 1 newline (`\n`) | Between fields within one email entry (sender, subject, action) |
| md | 2 newlines (`\n\n`) | Between email entries in the digest |
| lg | 3 newlines (`\n\n\n`) | Between account sections (Personal / Work) |

Exceptions: none. Telegram renders all spacing as newlines; no pixel-level control.

---

## Typography

Telegram controls all font rendering. The contract defines **semantic formatting** using HTML tags:

| Role | HTML Tag | Usage |
|------|----------|-------|
| Section header | `<b>PERSONAL</b>`, `<b>WORK</b>` | Account group headers |
| Category tag | `[UPPERCASE]` plain text | Category labels: `[CODE]`, `[CALENDAR]`, `[RESEARCH]`, `[HOME]`, `[URGENT]`, `[ROUTINE]`, `[SPAM]` |
| Email subject | Plain text in quotes | `"Server down alert"` |
| Reasoning text | `<i>Reasoning: ...</i>` | Chain-of-thought shown for low-confidence (<0.70) items only (D-07) |
| Action hint | `->` prefix, plain text | Recommended action: `-> Review PR` |
| Footer | `<i>Reply with a number...</i>` | Interaction instruction |

No custom font sizes or weights. Telegram renders `<b>` as bold and `<i>` as italic with platform-default fonts.

---

## Color

Telegram controls all colors. No hex values apply. The contract defines **visual emphasis hierarchy** using formatting and emoji markers:

| Role | Mechanism | Usage |
|------|-----------|-------|
| Dominant | Plain text (no formatting) | Email subject, sender, recommended action |
| Secondary | `<i>italic</i>` | Reasoning text, footer instructions, continued indicators |
| Accent/Alert | `[URGENT]` tag + standalone message | Urgent email notifications sent immediately (D-02) |
| Warning | Unicode marker in text | New/unknown sender flag (D-09) |

**Emphasis reserved for:**
- `<b>bold</b>`: Account headers only (PERSONAL, WORK)
- `[URGENT]` tag: Urgent classification only
- Unknown sender marker: First-time senders only (D-08, D-09)

## Copywriting Contract

### Digest Messages

| Element | Copy |
|---------|------|
| Account header (personal) | `<b>PERSONAL</b> ({N} new)` |
| Account header (work) | `<b>WORK</b> ({N} new)` |
| Continuation header | `<b>{ACCOUNT}</b> (continued)` |
| Digest footer | `<i>Reply with a number to act on that email.</i>` |
| Continuation indicator | `<i>continued...</i>` |

### Per-Email Entry Format

```
{number}. [{CATEGORY}] {sender}
   "{subject}"
   -> {recommended_action}
```

With optional suffixes:
- Unknown sender: append `\n   NEW SENDER` on its own line (D-09)
- Low confidence (<0.70): prefix action with `? ` and append reasoning:
  ```
  {number}. [{CATEGORY}] {sender}
     "{subject}"
     ? -> {recommended_action}
     <i>Reasoning: {chain_of_thought_summary}</i>
  ```
- Urgent: append ` URGENT` to category tag: `[CODE URGENT]`
- Multi-label: show primary category in tag; secondary categories noted in action text

### Urgent Standalone Notification (D-02)

```
URGENT EMAIL

From: {sender}
Subject: "{subject}"
Account: {personal|work}
Category: {category}

-> {recommended_action}

<i>Reply to act on this email.</i>
```

### Empty State

| Element | Copy |
|---------|------|
| No new emails (digest cycle) | `No new emails since last digest.` |
| No emails after spam filter | `{N} emails received, all filtered as spam/noise. No action needed.` |

### Error States

| Error | Copy |
|-------|------|
| Gemini API failure | `Classification unavailable -- Gemini API error. {N} emails queued for next cycle. Check API key and rate limits.` |
| Ollama not responding | `Local spam filter offline -- Ollama not responding. Sending all {N} emails to Gemini for classification.` |
| Gmail API failure | `Gmail sync failed for {account}. Error: {error_code}. Will retry next heartbeat cycle.` |
| Rate limit hit | `Gemini rate limit reached. {N} emails queued for classification in next cycle ({minutes} min).` |
| Batch size mismatch | (Internal only, not shown to user. Re-classify missing emails individually.) |

### Destructive Actions

Phase 2 has **no destructive actions**. Classification and digest delivery are read-only operations. Auto-archiving is deferred to Phase 4 (SPAM-03). No confirmation dialogs needed.

## Message Structure Constraints

| Constraint | Value | Source |
|------------|-------|--------|
| Max message length | 4000 characters (4096 limit minus 96 safety margin) | Telegram Bot API / RESEARCH.md Pitfall 3 |
| Parse mode | HTML | RESEARCH.md Pitfall 7 |
| Split strategy | Between email entries, never mid-entry | RESEARCH.md Pitfall 3 |
| Numbering | Continuous across split messages (1, 2, 3... not restarting per message) | TGRAM-04 requirement |
| Max entries per message | Variable -- depends on subject/sender length. Estimate ~15-20 entries per message. | Derived from 4000 char budget |
| Account grouping order | Work first, then Personal | Phase 1 D-12: work polled first |

## Interaction Patterns

### Numbered Selection (TGRAM-04)

User replies with a single number (e.g., `3`) to act on that email from the most recent digest.

| Input | Behavior |
|-------|----------|
| Valid number (in range) | Resolve to email via digest-map.json, present action options |
| Invalid number (out of range) | `No email #{N} in the last digest. Valid range: 1-{max}.` |
| Non-numeric reply | Ignored by digest handler (other OpenClaw handlers may process) |
| Reply to old digest (>24h) | `This digest has expired. Check the latest digest for current emails.` |

### Smart Batching (TGRAM-03)

| Priority | Delivery | Format |
|----------|----------|--------|
| Urgent (any confidence) | Immediate standalone notification | Urgent standalone template (above) |
| Non-urgent | Batched every 3 hours | Full digest format |
| Digest trigger | 3+ hours since last digest AND buffer has emails | Full digest format |
| Manual request | User asks for digest | Full digest format with all buffered emails |

### Confidence Tier Display (D-01)

| Tier | Confidence | Display Treatment |
|------|-----------|-------------------|
| Auto-label | >0.85 | Standard entry, no special markers |
| Act-and-confirm | 0.70-0.84 | Standard entry (silent confirm if user ignores per D-03) |
| Ask user | <0.70 | `? ` prefix on action line + reasoning shown in italic |

## Appendix: HTML Escaping Reference

All user-sourced strings (sender email, subject line) MUST be escaped before insertion into HTML-formatted Telegram messages:

```javascript
function escapeHtml(text) {
  return text
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;");
}
```
