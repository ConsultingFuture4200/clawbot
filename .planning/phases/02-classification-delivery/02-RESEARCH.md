# Phase 2: Classification & Delivery - Research

**Researched:** 2026-04-02
**Domain:** Email classification (Gemini structured output), Telegram digest delivery, local LLM pre-filtering
**Confidence:** HIGH

## Summary

Phase 2 turns raw Gmail polling data from Phase 1 into classified, prioritized email digests delivered via Telegram. The pipeline is: heartbeat detects new emails -> ollama/qwen2.5:7b spam gate -> Gemini 3 Flash structured JSON classification (batches of 5) -> digest formatting -> Telegram delivery with smart batching (urgent = immediate, low-priority = 3-hour batches).

The core technical challenges are: (1) configuring Gemini structured output with a well-defined JSON schema for 7-category multi-label classification, (2) managing Gemini free tier rate limits which are tighter than initially assumed (10 RPM for Flash on free tier, 250 RPD), (3) formatting Telegram digests within the 4096-character message limit with numbered selection for user interaction, and (4) building a sender cache from history.list data for unknown-sender detection.

**Primary recommendation:** Use `@google/genai` (v1.48.0) SDK with Zod schemas converted via `zod-to-json-schema` for type-safe classification. Batch 5 emails per Gemini call to stay well within rate limits. Use simple `fetch()` for ollama spam gate -- no SDK needed. Build sender cache incrementally from each polling cycle.

### Pattern 1: Two-Stage Classification Pipeline
**What:** Ollama spam gate (binary) -> Gemini 7-category classifier (structured JSON)
**When to use:** Every heartbeat cycle when new emails are detected
**Flow:**
```
1. Heartbeat detects new emails via history.list
2. Fetch metadata (sender, subject, snippet) for new messages
3. Pass each email through ollama spam gate:
   POST http://127.0.0.1:11434/api/generate
   { "model": "qwen2.5:7b", "prompt": "...", "stream": false }
   -> Returns "spam" or "not_spam"
4. Collect non-spam emails into batches of 5
5. For each batch, call Gemini with structured output:
   ai.models.generateContent({
     model: "gemini-3-flash",
     contents: prompt_with_few_shot_examples + batch_metadata,
     config: {
       responseMimeType: "application/json",
       responseJsonSchema: classificationSchema
     }
   })
6. Route results by urgency:
   - Urgent (any confidence): immediate Telegram notification
   - Low-priority: accumulate in batch-queue.json, send every 3 hours
   - Everything else: include in next digest cycle
```

const classificationItemSchema = z.object({
  email_index: z.number().describe("Index of the email in the input batch (0-4)"),
  categories: z.array(z.object({

  reasoning: z.string().describe("Chain-of-thought reasoning for the classification"),

### Pattern 3: Ollama Spam Gate (Lightweight)
**What:** Binary spam/noise filter using local qwen2.5:7b
**Key detail:** No npm package needed -- raw fetch to Ollama REST API
```javascript
async function spamGate(sender, subject, snippet) {
  const prompt = `You are a spam detector. Given an email's sender, subject, and snippet, respond with ONLY "spam" or "not_spam".

Sender: ${sender}
Subject: ${subject}
Snippet: ${snippet}

Classification:`;

  const res = await fetch("http://127.0.0.1:11434/api/generate", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      model: "qwen2.5:7b",
      prompt,
      stream: false,
    }),
  });

  const data = await res.json();
  const answer = data.response.trim().toLowerCase();
  return answer.includes("spam") && !answer.includes("not_spam") ? "spam" : "not_spam";
}
```

### Pattern 4: Telegram Digest Formatting
**What:** Structured digest with account grouping, priority sections, numbered selection
**Key constraints:** 4096 chars per message, HTML parse mode, numbered items for reply-selection
```javascript
function formatDigest(classifiedEmails, account) {
  // Group by priority tier
  const urgent = classifiedEmails.filter(e => e.primary_category === "urgent");
  const normal = classifiedEmails.filter(e =>
    !["urgent", "spam_noise"].includes(e.primary_category) && e.primary_confidence >= 0.70);
  const lowConf = classifiedEmails.filter(e => e.primary_confidence < 0.70);

  let msg = `<b>${account.toUpperCase()} INBOX</b>\n\n`;
  let itemNum = 1;

  if (urgent.length > 0) {
    msg += `<b>URGENT</b>\n`;
    for (const e of urgent) {
      const newSender = e.isUnknownSender ? " NEW" : "";
      msg += `<b>${itemNum}.</b> ${escapeHtml(e.sender)}${newSender}\n`;
      msg += `   <i>${escapeHtml(e.subject)}</i>\n`;
      msg += `   ${e.primary_category} | ${e.recommended_action}\n\n`;
      itemNum++;
    }
  }

  // ... similar blocks for normal and lowConf items
  // Low-confidence items include reasoning summary (D-07)
  for (const e of lowConf) {
    msg += `<b>${itemNum}.</b> ${escapeHtml(e.sender)}\n`;
    msg += `   <i>${escapeHtml(e.subject)}</i>\n`;
    msg += `   ? ${e.primary_category} (${Math.round(e.primary_confidence*100)}%)\n`;
    msg += `   <i>Why: ${e.reasoning}</i>\n\n`;
    itemNum++;
  }

  msg += `\nReply with a number to act on that email.`;
  return splitMessage(msg, 4096);  // Split at item boundaries
}
```

### Pattern 5: Sender Cache
**What:** Per-account JSON file tracking known senders for "unknown sender" detection
```javascript
// sender-cache-personal.json structure
{
  "senders": {
    "alice@example.com": { "first_seen": "2026-04-01T10:00:00Z", "count": 5 },
    "newsletter@company.com": { "first_seen": "2026-03-15T08:00:00Z", "count": 42 }
  },
  "last_updated": "2026-04-02T12:00:00Z"
}

// Check and update
function isUnknownSender(senderEmail, cacheFile) {
  const cache = JSON.parse(fs.readFileSync(cacheFile, "utf8"));
  if (cache.senders[senderEmail]) {
    cache.senders[senderEmail].count++;
    fs.writeFileSync(cacheFile, JSON.stringify(cache, null, 2));
    return false;
  }
  // New sender -- add to cache
  cache.senders[senderEmail] = { first_seen: new Date().toISOString(), count: 1 };
  cache.last_updated = new Date().toISOString();
  fs.writeFileSync(cacheFile, JSON.stringify(cache, null, 2));
  return true;
}
```

### Anti-Patterns to Avoid
- **Sending full email body to Gemini for classification:** Only metadata (sender, subject, snippet) is needed. Full body wastes tokens and hits rate limits faster.
- **Calling Gemini once per email:** Batch 5 at a time per D-05. Individual calls burn 5x the RPM quota.
- **Using ollama for full classification:** qwen2.5:7b is only for binary spam gate. Multi-category classification requires Gemini's structured output quality.
- **Storing email content in state files:** Privacy constraint from AGENTS.md -- store metadata references only, never email body text.
- **Blocking on mid-confidence confirmations:** D-03 says ignored = implicitly confirmed. Never block the pipeline waiting for user response.

## Common Pitfalls

### Pitfall 1: Gemini Free Tier Rate Limits Are Tighter Than Expected
**What goes wrong:** Classification fails with 429 errors during busy email periods
**Why it happens:** Gemini free tier for Flash is approximately 10 RPM and 250 RPD (reduced in Dec 2025). With batches of 5 emails, 250 RPD means ~1250 emails/day max classification capacity -- adequate but not unlimited.
**How to avoid:** Track RPM and RPD usage in state. If approaching limits, increase batch size or defer non-urgent classification. Log quota consumption per heartbeat cycle.
**Warning signs:** 429 responses from Gemini API, classification backlog growing

### Pitfall 2: Telegram 4096-Character Limit Mid-Item Splits
**What goes wrong:** Digest message gets truncated or splits in the middle of an email item, breaking formatting
**Why it happens:** Naive character-count splitting doesn't respect item boundaries
**How to avoid:** Build digest items as discrete units. Track cumulative length. When adding next item would exceed limit, start a new message. Include page indicators ("1/3", "2/3", etc.).
**Warning signs:** Garbled messages in Telegram, missing closing HTML tags

### Pitfall 3: Ollama Not Running When Heartbeat Fires
**What goes wrong:** Spam gate fails, entire classification pipeline halts
**Why it happens:** Ollama service not started on WSL2 boot, or GPU memory exhausted
**How to avoid:** Wrap ollama fetch in try/catch with timeout. If ollama is unreachable, skip spam gate and send all emails to Gemini (graceful degradation). Log the skip.
**Warning signs:** ECONNREFUSED on 127.0.0.1:11434, classification delays

### Pitfall 4: Stale historyId Causing Duplicate Processing
**What goes wrong:** Same emails get classified multiple times after a 404 on history.list
**Why it happens:** historyId expires (Gmail retains ~30 days). Full resync fetches already-processed messages.
**How to avoid:** Maintain a "processed message IDs" set in state file (bounded to last 1000 IDs). On history.list 404, do full sync but skip already-processed messages.
**Warning signs:** Duplicate items in Telegram digests, same email appearing multiple times

### Pitfall 5: Few-Shot Examples Mismatch With Schema
**What goes wrong:** Gemini returns inconsistent or malformed classification despite schema
**Why it happens:** Few-shot examples in the prompt don't match the response schema structure
**How to avoid:** Few-shot examples in classification-examples.json must use the EXACT same JSON structure as the responseJsonSchema. Validate examples against the Zod schema at startup.
**Warning signs:** JSON parse errors on Gemini responses, missing fields

### Pitfall 6: Batch Queue Timer Drift
**What goes wrong:** Low-priority batch never sends, or sends too frequently
**Why it happens:** 3-hour batch timer implemented with in-memory setTimeout resets on gateway restart
**How to avoid:** Store `next_batch_send_time` in batch-queue.json. On each heartbeat, check if current time >= next_batch_send_time. This survives restarts.
**Warning signs:** Users not receiving low-priority digests, or receiving them at inconsistent intervals

### Message Splitting for Telegram
```javascript
// Source: Telegram Bot API 4096-char limit
function splitMessage(htmlText, maxLen = 4096) {
  if (htmlText.length <= maxLen) return [htmlText];

  const messages = [];
  const items = htmlText.split(/(?=<b>\d+\.<\/b>)/); // Split before numbered items
  let current = "";

  for (const item of items) {
    if ((current + item).length > maxLen - 20) { // 20 chars reserved for page indicator
      if (current) messages.push(current.trim());
      current = item;
    } else {
      current += item;
    }
  }
  if (current.trim()) messages.push(current.trim());

  // Add page indicators
  return messages.map((msg, i) =>
    `${msg}\n\n<i>(${i + 1}/${messages.length})</i>`
  );
}
```

### Smart Batching Timer Logic
```javascript
// Source: Project decisions D-02, TGRAM-03
const BATCH_INTERVAL_MS = 3 * 60 * 60 * 1000; // 3 hours

async function routeClassifiedEmail(classified, account, stateDir) {
  const queuePath = path.join(stateDir, "batch-queue.json");
  const queue = JSON.parse(fs.readFileSync(queuePath, "utf8"));

  // Urgent: always send immediately (D-02)
  if (classified.categories.some(c => c.category === "urgent")) {
    await sendImmediateTelegramAlert(classified, account);
    return;
  }

  // Non-urgent: add to batch queue
  queue.items.push({ ...classified, account, queued_at: Date.now() });

  // Check if batch timer has elapsed
  if (Date.now() >= queue.next_batch_send_time) {
    await sendBatchDigest(queue.items, account);
    queue.items = [];
    queue.next_batch_send_time = Date.now() + BATCH_INTERVAL_MS;
  }

  fs.writeFileSync(queuePath, JSON.stringify(queue, null, 2));
}
```
